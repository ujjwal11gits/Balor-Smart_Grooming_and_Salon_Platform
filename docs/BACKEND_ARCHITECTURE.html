Balor — Backend System Architecture & Design Guide

This guide details the complete backend implementation of the Balor grooming booking system, detailing all design patterns, directory configurations, security systems, database designs, and core business logics.

1. Folder Architecture & Entry Points

The backend follows a modular, RESTful design built with Node.js, Express.js, and Mongoose.

Directory Structure

backend/

│   └── auth.js              # Authentication and authorization guards (JWT + Role Checks)
├── models/
│   ├── User.js              # User details, passwords, dynamic OTPs and roles
│   ├── Salon.js             # Salon schema containing subdocuments for services and barbers
│   ├── Barber.js            # Barber schemas, ratings, working slots, and leaves

│   ├── bookings.js          # Booking reservations, cancels, and OTP-based completions

Entry Points Explanation

  • server.js: Initializes the database, loads global security tools (Helmet, CORS, cookie parsers), registers API routing controllers, and contains the fallback global error handling middleware.
  • dev-server.js: Imports the configured server and listens on `PORT` dynamically, allowing hot-reloads using nodemon.

2. Security Hardening & Middlewares

A. HTTP Headers Hardening (helmet)

app.use(helmet());

B. Restricted Cross-Origin Requests (CORS)

app.use(cors({
  origin: process.env.FRONTEND_URL || 'http://localhost:5173',
  credentials: true // Crucial: Permits secure browser cookie transmissions
}));

C. Authentication Rate Limiter (express-rate-limit)

const authLimiter = rateLimit({
  windowMs: 15 * 60 * 1000, // 15 minutes
  max: 30, // Limit each IP to 30 requests per window
  message: { message: "Too many authentication requests, please try again after 15 minutes." }
});

D. JWT Verification Middleware (middleware/auth.js)

module.exports = (allowedRoles = []) => {
  return (req, res, next) => {
    const authHeader = req.headers.authorization;
    if (!authHeader || !authHeader.startsWith('Bearer ')) {
      return res.status(401).json({ message: 'Access token missing' });
    }
    const token = authHeader.split(' ')[1];
    try {
      const decoded = jwt.verify(token, process.env.JWT_SECRET);
      req.user = decoded; // Contains id, email, and role
      
      // If role checks are defined, enforce them
      if (allowedRoles.length && !allowedRoles.includes(decoded.role)) {
        return res.status(403).json({ message: 'Forbidden: Access denied' });
      }
      next();
    } catch (err) {
      return res.status(401).json({ message: 'Access token expired or invalid' });
    }
  };
};

E. Global JSON Error Formatting

Why we do this:

Prevents default HTML stack trace leaks which expose folder trees and database variables to potential attackers.

app.use((err, req, res, next) => {
  console.error(err.stack);
  res.status(500).json({
    message: 'An unexpected internal server error occurred.',
    error: process.env.NODE_ENV === 'development' ? err.message : undefined
  });
});

3. Database Schema Design & Indexing

A. Database Indexing (Performance Optimization)

To avoid collection scans, we apply B-Tree indexes to all foreign key lookups ($O(\log N)$ instead of $O(N)$ lookup complexity):

bookingSchema.index({ userId: 1 });
bookingSchema.index({ barberId: 1 });
bookingSchema.index({ salonId: 1 });
bookingSchema.index({ date: 1 });
barberSchema.index({ salonId: 1 });
salonSchema.index({ ownerId: 1 });
waitlistSchema.index({ barberId: 1, date: 1 }); // Compound Index

4. Core API Routes & Business Logic

A. Auth Flow & Session Management (HttpOnly Cookie Rotation)

Splits JWT tokens: Generates short-lived Access Tokens returned in JSON payload for client consumption, and long-lived Refresh Tokens mapped strictly inside secure, HTTP-only cookies to prevent XSS-based hijacking.

B. Dynamic Time-Slot Calculations

Checks requested booking dates against barber configuration slots and leaves, scans active database bookings, dynamically blocks off target duration sizes, and filters out already passed slot values relative to current timezones.

C. Rating Recalculation & Salon Subdocument Sync

Submitting a review triggers an aggregate recalculation of the barber's average rating. Mongoose immediately updates the Barber collection, and updates the embedded barber profile subdocument array inside the parent Salon document, keeping landing page queries fast.

5. Backend Technical & System Design FAQ

Q1: Why do you store the Refresh Token in an HttpOnly cookie instead of LocalStorage?

Answer: LocalStorage is vulnerable to XSS. If a script injection exploit occurs, malicious code can steal the keys. HttpOnly cookies cannot be read by browser JavaScript, blocking script-based theft.

Q2: Explain how the backend calculates wait times.

Answer: It counts active bookings scheduled for today that are pending or confirmed up to the current time, multiplying that count by an average service time multiplier (30 minutes).

Q3: What are MongoDB subdocuments and why did you use them in the Salon schema?

Answer: Subdocuments are schemas nested inside a parent document. In this project, `Salon` embeds arrays of `services` and `barbers`. We used this because these arrays are queried on every detail page load; embedding them gets all details in a single database lookup instead of doing multiple collection joins.
