Balor: System Architecture Guide

A detailed explanation of the frontend and backend architectures, database modeling, and communication protocols.

📐 1. System Overview

The **Balor Salon Platform** is built using a decoupled Client-Server architecture. The React frontend handles the presentation, state management, and user interaction, while the Node.js/Express backend handles business logic, security, scheduling algorithms, and database operations.

[Browser / Client]
      │
      ▼ (HTTPS / JSON API Requests)
[Nginx Web Server / Reverse Proxy]
      ├── (Port 80/443: Serves React Static Files)
      └── (Port 5000 /api: Proxies to Node.js Backend)
              │
              ▼
      [Node.js Express App]
              ├── (Mongoose ODM) → [MongoDB Atlas]
              └── (Nodemailer SMTP) → [Gmail SMTP]

🎨 2. Frontend Architecture (React / Vite)

The frontend is a Single-Page Application (SPA) designed for speed, using component-driven development.

A. Project Directory Structure

  • /src/api/: Contains global configuration for HTTP clients (Axios).
  • /src/components/: Reusable UI layout elements (Navbar, Modals, Cards, Buttons).
  • /src/context/: Global State Management (AuthContext handles user logins, roles, and cookies).
  • /src/pages/: Role-specific sub-dashboards (Customer, Barber, Owner, Admin).
  • /src/utils/: Shared helper functions (dates formatting, validation).

JWT Access Token Handling (Axios Interceptors)

To avoid security vulnerabilities, access tokens are stored in memory and attached automatically to requests. If a request returns a 401 Unauthorized (token expired), a custom Axios interceptor automatically hits the backend /auth/refresh endpoint to fetch a new token via HTTP-only cookies and retries the original request seamlessly.

⚙️ 3. Backend Architecture (Node.js / Express)

The backend is modular, relying on the classic Controller-Route-Model pattern to ensure clean separation of concerns.

A. Layered Responsibility

  • Router Layer (routes/): Parses incoming URLs and assigns them to controller actions.
  • Controller Layer (business logic): Contains functions executing specific actions (e.g. booking a slot, registering a shop).
  • Middleware Layer:
    • authMiddleware.js: Verifies JWT tokens and checks user roles (Customer, Owner, Barber, Admin).
    • errorHandler.js: Catches exceptions and returns standardized JSON errors.
  • Model Layer (models/): Connects directly to MongoDB, enforcing schemas and data relationships.

💾 4. Database Modeling & Relationships

Although MongoDB is NoSQL, our platform relies on relational model binding using Mongoose Object IDs (ref):

[User (Customer, Owner, Barber)]
   │ 1         ▲ 1
   ▼ N         │ N
[Booking] → [Barber] → [Salon]

Data Binding Flow:

  • Users Schema: Stores credentials, name, email, contact, and role.
  • Salon Schema: Owned by a User (Owner role), contains working hours, address, and list of offered services.
  • Barber Schema: Bound to a Salon (Object ID reference) and belongs to a User (Barber role), keeping track of their personal schedule.
  • Booking Schema: Relates a Customer (User Object ID), a Salon, a selected Barber, and specific services, recording start/end timings and status.

📧 5. Integration Services

  • Secure Authentication: Double JWT token authentication (Short-lived access token + long-lived secure HttpOnly refresh cookies) preventing XSS and CSRF attacks.
  • Transactional Emails: Integrates Nodemailer utilizing Gmail secure SMTP relays to fire booking confirmation emails, cancellation alerts, and owner notifications.
